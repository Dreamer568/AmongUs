 Why Proximity Voice Chat is Failing
1. Microphone Access Race Condition (No Audio Tracks Added)
In your lobby setup, when a player clicks "Create Room" or "Join Lobby", initLocalVoice() is called, which immediately triggers the browser's microphone permission prompt:
code
JavaScript
initLocalVoice();
socket.emit('create_room', ...);
Because initLocalVoice() is an asynchronous function requesting hardware permissions, the browser takes several seconds to display the prompt and wait for the user to click "Allow."
In the meantime, the script immediately continues and emits create_room or join_room. When the peer connections are set up shortly after, localAudioStream is still null. As a result, the following check fails:
code
JavaScript
if (localAudioStream) {
  localAudioStream.getTracks().forEach(track => pc.addTrack(track, localAudioStream));
}
Because localAudioStream was not yet available, no audio tracks are attached to the RTCPeerConnection when it is negotiated. Even if the user allows the microphone a second later, the connection is already established with zero audio tracks.
2. Peer Connections are Never Initiated at Game Start
In your code, peer connections are only initiated in socket.on('player_joined') if the main menu overlay is already hidden:
code
JavaScript
if (mainOverlay.style.display === 'none') {
  socket.emit('initiate_peer_connection', { targetId: p.id });
}
When players join the pre-game lobby, the menu is still visible, so they skip this connection step. When the host eventually starts the game and game_started is fired, the menu is hidden, but no connection signaling is triggered for the players who were already in the lobby. Only mid-game joiners trigger connections, leaving the initial lobby players unable to hear each other.
How to Fix the Issue
To resolve these issues:
Make the room creation and joining handlers async so they await microphone acquisition before registering on the socket server. This guarantees that your microphone stream is fully ready before any WebRTC peer negotiation starts.
Trigger the signaling process when the match begins in the game_started event.
Use a lexicographical ID tie-breaker (socket.id < p.id) to ensure exactly one side initiates the connection, preventing redundant connection negotiations.
Here are the specific updates needed in your client-side code:
